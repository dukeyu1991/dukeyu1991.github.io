python classmethod和staticmethod区别

发表于 2016-09-29

其实classmethod和staticmethod没有什么特别大的区别。
但还是需要注意如下几点：

  1. staticmethod函数参数没有要求，而classmethod函数第一个参数必须传类(该参数名通常使用cls)。

  2. 调用的方法其实类似,实际上classmethod是先获取了实例的类，再调取该方法。如果直接使用类调用，则无需进行获取类。

python pdb入门

发表于 2016-08-24

之前一直用print去调试Python代码，经常左一个print，右一个print。偶尔碰上去查询大量数据的代码，不仅麻烦，而且很浪费时间。整体的调试效率很低。为了提高调试的效率，决定学习一下pdb。
pdb提供了在Python debug模式中交互的功能，在交互的过程中包含这各个变量，非常方便的去调用。并且能够一步一步的去执行代码，这样能够更加清晰的认识程序实际的执行过程并找出其中的bug。

启动pdb

首先，需要在想要调试的代码处启用调试交互。这里有几种方式去启动pdb

  2. 在Python交互环境里 In[3]中的main为pdb_test的函数

  3. 在代码需要调试处。在需要进行调试的代码前加入 pdb.set_trace(), 然后启动命令行

  4. 程序报错后

pdb主要的命令

进入了调试模式后，可以使用 h(elp) 查看帮助。以下将pdb主要的几个命令罗列在下，以备不时之需查用。

断点设置

  1. b(reak) [代码行号] | [文件名:代码行号] 不加数字列出所有断点，加数字表示在行号处增加一个断点。

  2. disable 断点编号 令断点失效，但不同cl删除断点。它只让断点失效，但不删除。

  3. enable 断点编号 将3中失效的断点重新置为有效。

  4. tbreak [代码行号] | [文件名:代码行号] 临时性的断点，当该短点首次被击中时，会自动被移除。其他的同break没有区别。

  5. commands 断点编号在断点前执行前，可以启动一个脚本编辑页，在其中显示想要或需要显示的变量，输入end退出编辑页。然后再执行该行前，使用continue，就能看到之前编辑过的显示的变量。

程序运行

  1. n(ext) 单步运行

  2. s(tep) 进入函数内

  3. c(ont(inue)) 调到下一个断点

  4. r(eturn) 执行代码直至程序从当前的函数返回

  5. unt(il) 当执行的代码大于当前执行的位置行号, 停止。通常用于等待一个遍历或者重复的代码段执行完毕。

  6. j(ump) 代码行号 直接跳到目标代码行执行。方便你重新设定需要执行的位置，也可以略过不需要执行的代码。但是不能跳入一个循环中或者一个try:except的代码块中。finally的代码必须要执行，所以也不能跳出finally的代码段。

  7. u(p) 从当前的未知跳到执行的栈中的上一层。

  8. d(own) 作用与up相反，从当前的栈层中跳入下一层的栈中。

  9. ignore 断点编号 [忽略次数] 忽略某个断点n次。如果没有忽略次数，将该断点重新设置为有效，不会被忽略。
  10. condition 断点编号 [判断条件]设置某个断点在判断条件为True时，断点才有效。省略判断条件，则令之前condition设定的所有条件失效。

程序查看

  1. w(here) 显示当前代码所处的栈层
  2. l(ist) 会显示当前执行的附件的代码，不断的使用list会不断列出代码，直至代码末尾。

  3. a(rgs) 显示当前函数所有的变量

  4. p 表达式 显示 表达式 在当前环境中执行的的结果

  5. pp 表达式 功能同4，但是使用Python中的pprint模块

  6. whatis 表达式 显示表达式运算结果的类型

  7. bt 貌似和where一个功能，都是显示栈层的。

pdb自身命令

  1. q(uit) 退出调试环境。
  2. exit 作用同1。

  3. alias [别名 [命令]] 将某一个命令起一个别名。执行该别名，相当于执行命令。后面不带参数，表示查看所有的关联的命令。

  4. unalias 别名 作用和 alias相反，取消命令和别名关联。

  5. run 当用命令行方式启动的调试时,执行run能够重新启动调试的代码。

  6. restart 就是 run的一个马甲。
  7. debug在debug里面再开一个debug界面，相当于递归debug。

写在最后

现在好多编辑器都集成了pdb debug功能，界面更加的友好。譬如pycharm就有很好的debug功能。在IDE debug的时候，代码显示的更多，可以更加方便的调试。

version_1中产生问题的分析

在分析问题前，我们先需要了解几个基本的概念

  1. 递归深度
    在Python函数执行时，遇到中断，系统会把正执行的环境保存下来，压入栈中。在version_1中，执行到入栈那行时，系统栈中就会增加一个。而递归深度是指在执行该函数时，系统栈压入对象的个数。
  2. Python最大的递归深度
    Python解释器中规定了栈的最大深度。程序压入栈的对象超过了栈的深度，就会发生栈满。从而导致栈溢出的错误。

    version_1栈溢出分析

    通过上面的基本概念后，我们可以去分析version_1中为何会出现溢出的错误。return fib(n - 1) + fib(n - 2),系统将此时的环境压入栈，转而去执行fib(n - 1)。类似的在执行 return fib(n - 2) + fib(n - 3), 将fib(n - 1)执行环境压入栈中。依次类推，系统栈不断增加，最终导致栈溢出。

通过尾递归尝试解决递归栈溢出问题

尾递归是指:

  1. 在函数返回的时候，调用自身本身。
  2. return语句不能包含表达式。
    能满足上述两条规则，编译器或者解释器能够进行优化，使得递归本身就占用一个栈空间。无论多少次调用，都不会导致栈溢出

python编码问题

发表于 2016-08-04

在Python编程中，会经常在decode, encode的时候出现各种问题，诸如
UnicodeDecodeError: ‘ascii’ codec can’t decode byte 0xe4 in position 0: ordinal not in range(128)
UnicodeEncodeError: ‘ascii’ codec can’t encode characters in position 0-1: ordinal not in range(128)
可是到底是什么原因导致了这个问题呢？
其中的原因如果没能完全搞清楚，那在未来的日子里想必也是超级痛苦的。

原因分析

众所周知，计算机存储各式文件，无论是视频、音频、图片、文字，无非就是在硬盘中保存了一长串的01字符。可是计算机在运行时是通过什么样的方式去表示文字的呢？

ASCII码

美帝在二战时期发明了第一台电脑。理所应当，计算机的一切表示方式都是按照美帝人民的习惯来的。
英文总共不过26个字母，在加上一些符号，还有控制符，统共不过100+个字符。一个字节8位能有256中状态，用一个字节去表示足够了。聪明的科学家制定了一套字符编码标准，ASCII码，每个字符对应一串数字。最初ASCII码总共有128个字符编码，后来发现不够用，就出现了可扩展的ASCII码。
对ASCII码有兴趣的同学可自行Google，这里的ASCII码对本文已经够了

Unicode

随着电脑进入了世界各地的千家万户，电脑的使用者不仅仅是英文国家的民众。举个栗子，在我大中国，常用汉字都有6000+了。这时，字符已经远远的超越了ASCII码表示范围了。再加上韩国、日本等世界其他国家的字符，毛估估有几十万个。如此庞大的字符库该如何表示呢？
这时候有全球标准化组织提出了unicode编码，用2个字节去表示字符。2个字节总共能表示65536个字符，而这个数字还是远远不够。因此更近一步，在后面的规范中定义了一组附加字符，它也是由2个字节表示。也就是unicode总共有4个字节。理论上已经足够去涵盖一切文字的符号了。一旦字符的unicode编码确定下来就不会再改了。
但unicode有一个很大的缺陷，太浪费空间了。一个字符’A’用一个字符就足够表示了，剩下的三字节就完全浪费掉了。正因为这个原因，unicode有各种编码方式：utf-8、utf-16、gbk等等。

以上的背景就足够继续接下来的问题分析了，如果有兴趣深入的，请自行Google。

git分支管理

发表于 2016-06-17

工作时，git使用是没有清晰的流程，往往会造成在提交代码时经常和线上的代码库冲突。更加严重的情况，为了想维持git log的线性状态而使用rebase，但不是完全了解rebase的功能。遇到冲突时，merge和rebase就乱抓乱用，让git log变得一团糟。真正遇到问题时，根本不太可能通过git log分析出问题出现的位置。
Google了许多的文章后，我学到了很多，以后自己就这样用git了。

具体的工作流程

  1. 假设你现在在master分支

  2. 在你自己的工作分支上码代码ing， 完成后

  3. 返回你当前分支的父分支，并且保持此分支的远端和本地保持一致

  4. 返回你的工作目录，并且将工作分支保持为最新

  5. 再返回master分支上，将新的需要提交的commit合并到master中，并且推到远端master上。最后，删除你的工作分支

这样能保证提交的git log维持在一个线性的有序的风格，便于分析如：代码提交、出错分析等各种问题。
